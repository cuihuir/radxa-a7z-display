# 决策日志

## 使用方式

把需要长期保留的项目决策记录在这里，内容要短而明确。

## 条目格式

- 日期
- 决策
- 原因
- 影响

## 当前决策

### 2026-07-06：仓库语言策略

- 决策：以英文为源文本，所有核心文档都要有中文翻译。
- 原因：项目需要长期维护，因此必须保留稳定的双语记录。
- 影响：任何重要文档变更都必须同时更新两个语言版本。

### 2026-07-06：项目身份

- 决策：项目名称使用 `radxa-a7z-display`。
- 原因：名称足够具体、便于检索，并且与项目范围一致。
- 影响：仓库名、文档和未来的 GitHub remote 都应使用这个身份。

### 2026-07-06：源码对比工具范围

- 决策：先做一个小型对比工具，用来检查厂商源码树并输出 Markdown 差异报告。
- 原因：第一步最有用的代码工作，是帮助比较板级配置、家族配置和 DTS 文件，而不是先把下载或构建自动化拉进来。
- 影响：初版实现保持小、可测试，并且能直接服务 A733 移植分析。

### 2026-07-06：树检查子命令

- 决策：在对比工具里增加单树验证模式，用来做 A733 源码树的最小检查。
- 原因：一个轻量的源码树检查器，比先做下载器或构建流水线更实用。
- 影响：这个工具现在既能回答“哪里不一样”，也能回答“这棵树是否至少具备基础检查条件”。

### 2026-07-06：Radxa 源码树目标

- 决策：把 `rsdk` 视为 Radxa A733 的真实构建源码，把 `radxa-cubie-a7z` 继续当成发布/工作流外壳。
- 原因：Radxa 的公开产品仓库只指向 builder，真正的 A733 策略和包逻辑在 `rsdk` 里。
- 影响：源码对比和移植分析应使用 `rsdk` 对比 Orange Pi 的构建树。

### 2026-07-06：A7Z Debian 12 报告生成器

- 决策：增加一个专门面向 A7Z Debian 12 迁移的报告生成器，读取本地 Radxa 和 Orange Pi 源码树。
- 原因：主目标是把 `radxa-cubie-a7z` 推到 Debian 12 桌面，因此工具应把源码信号转成可执行的移植简报。
- 影响：仓库现在有了一条真正服务移植决策的代码路径，而不只是做树对比。

### 2026-07-06：Radxa A733 Bookworm 试验分支

- 决策：在本地 Radxa `rsdk` 试验分支里为 `radxa-a733` 暴露 `bookworm`，同时保留现有的 `kde` 默认值。
- 原因：实际构建路径需要先有 Debian 12 的入口，才能开始跑板级验证。
- 影响：`radxa-a733` 现在本地默认指向 `bookworm`，下一步是做真实的构建/启动测试。

### 2026-07-13：不要对 A733 发布镜像使用 PiShrink

- 决策：发布时压缩已经验证的原始镜像，可使用 XZ 或 Zstandard；不要运行 PiShrink 或其他会改写 GPT 的工具。
- 原因：原始 RSDK `output.img` 可以在 A7Z 启动，而 PiShrink 产物不能。PiShrink 改变了根分区 UUID，并移除了其 `LegacyBIOSBootable` GPT 属性。
- 影响：`v0.1.0-a733-debian12-kde` 不再作为可安装镜像发布。替代版本必须逐字节保持原始分区表，并在发布前完成烧录和启动验证。

### 2026-07-14：恢复按 EDID 原生时序选择 HDMI

- 决策：移除 A733 强制 FHD/最大 60Hz 的 DRM 策略，并在首项 fallback 前优先选择 DRM preferred mode。
- 原因：该策略把 `FLY-HDMI-LCD7` 的原生 `1024x600` 强制为 `1920x1080`，造成拉伸和裁切。
- 影响：已验证的稳定部署是原厂 `5.15.147-21-a733` 内核加 A7Z DTB 修复；后续完整内核包不得并发重建 DKMS。

### 2026-07-14：保持 A7Z initramfs 接近厂商镜像大小

- 决策：在显示内核包中加入 initramfs hook，移除早期 fsck 程序及仅供 ext2fs 使用的库。
- 原因：内容有效但达到 42.67 MB 的 initramfs 在持久化内核日志前启动失败，而相同 kernel、DTB、模块和命令行配合 42.25 MB initramfs 可以启动。
- 影响：`+display2` 包生成 42,251,173 字节的 initramfs 并已成功启动；系统根文件系统中的正常及离线修复工具不受影响。

### 2026-07-21：早期启动失败必须以串口证据为准

- 决策：不能只根据 HDMI 黑屏或 SSH 超时判断 A7Z 启动失败；必须用 UART0、`115200 8N1` 捕获从内核启动到登录提示的日志。
- 原因：精确为 42,251,056 字节的 initramfs 完成了本次有串口记录的 `l0` 启动。更早的故障还包含已确认的 ext4 损坏和启动项破坏，因此只看 SSH 超时无法区分网络恢复慢和真实启动失败。
- 影响：后续 kernel/initramfs 实验必须保留 `l1`，记录串口日志，并在修改 SD 卡前先确定明确的 UART 失败边界。

### 2026-07-21：放弃 `MODULES=dep` 早期 fsck initramfs

- 决策：保留已验证的完整模块 initramfs 和现有离线 rootfs 修复路径；不发布实验性的 `MODULES=dep` 加早期 `e2fsck` 配置。
- 原因：14 MB 镜像包含预期 fsck 工具，但此前测试没有 UART 证据，因此没有建立可靠的失败边界。
- 影响：仓库恢复到 `display3` initramfs 策略；实验镜像只能作为隔离的 U-Boot 启动项并配合串口重新测试；新增持久 postinst hook，在 `u-boot-update` 后自动修复独立 `l1` 的 DTB 和 PowerVR blacklist。

### 2026-07-21：将早期 fsck 保留为仅恢复候选方案

- 决策：14 MB 的 `fsck-final-test` initramfs 已确认可以启动，但只作为隔离的恢复候选项，不替换正常的 `display3` initramfs。
- 原因：UART 日志证明 U-Boot 成功加载 14,701,871 字节镜像，initramfs 在挂载根分区前对 `/dev/mmcblk0p3` 执行 `fsck.ext4 -a`，报告文件系统干净，并最终到达 `ttyAS0` 登录提示。这推翻了此前缺少串口证据时“小镜像无法启动”的假设，但尚未证明它能修复人为构造的文件系统损坏。
- 影响：继续以 `l0` 作为已知可用默认项，离线 `e2fsck` 仍是受支持的修复方法。未来若打包恢复启动项，必须保持可选、在 UART 上输出详细日志，并先通过可复现的文件系统故障测试。
