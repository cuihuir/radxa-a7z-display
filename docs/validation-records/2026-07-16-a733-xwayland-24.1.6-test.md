# A733 XWayland 24.1.6 Glamor Test

## Objective

Determine whether a newer XWayland fixes the software glamor fallback while
KWin uses the proprietary PowerVR renderer and `card0` remains the HDMI KMS
device.

## Build

XWayland `24.1.6`, matching Debian 13/Trixie, was built natively on the A7Z
against Debian 12 plus Bookworm backports:

- XWayland source SHA-256:
  `737e612ca36bbdf415a911644eb7592cf9389846847b47fa46dc705bd754d2d7`;
- locally installed xorgproto `2024.1` headers;
- libdrm `2.4.123`, Mesa/GBM `25.0.7`, wayland-protocols `1.43`;
- single-threaded build with a temporary 2 GiB swap file;
- resulting ARM64 binary SHA-256:
  `a58c2908da43d45bdf28f63b692c260452b9f455ce9aa36cd52a33190f389135`.

The binary was installed only under `/home/radxa/xwayland-build/prefix`; the
Debian `xwayland 22.1.9-1` package was never replaced.

## Results

| XWayland | Graphics environment | Result |
|---|---|---|
| `22.1.9` | Distribution Mesa | `egl: failed to create dri2 screen`, software fallback |
| `22.1.9` | PowerVR DRI/EGL | Loads `pvr_dri.so`, then software fallback |
| `24.1.6` | Distribution Mesa | `EGL setup failed, disabling glamor` |
| `24.1.6` | PowerVR DRI/EGL | `Main linux-dmabuf device has no render node` |

In every case XWayland opened both `/dev/dri/card0` and
`/dev/dri/renderD128`, so the failure is not caused by a missing file
descriptor or permissions.

## Multi-GPU Device-Order Test

KWin was also tested with
`KWIN_DRM_DEVICES=/dev/dri/card1:/dev/dri/card0`. This made the PowerVR node the
primary DRM device but failed because `card1` is not a complete KMS device:

- atomic modesetting capability failed on `card1`;
- KWin could not create its linux-dmabuf feedback table;
- cross-GPU AR24 framebuffer creation returned `EINVAL`;
- XWayland exited and Plasma startup timed out.

This configuration was immediately reverted over SSH.

## Conclusion

Updating XWayland alone does not enable glamor. The current topology has a KMS
display device (`card0`) without a render node and a proprietary render device
(`card1`/`renderD128`) without the KMS behavior KWin expects from a primary DRM
GPU. The next useful work is to inspect or patch KWin's linux-dmabuf feedback
selection for this split topology, while keeping `card0` as the primary KMS
device.

The board was restored to XWayland `22.1.9`, `gpu6` clean wrappers,
`default l0`, and the normal SDDM login screen. Temporary autologin and swap
files were removed.

## Render-Node Feedback Follow-Up

On 2026-07-17, KWin's linux-dmabuf feedback was temporarily overridden to
advertise the EGL render node `/dev/dri/renderD128` while retaining
`/dev/dri/card0` for KMS. XWayland `24.1.6` then passed its previous
`Main linux-dmabuf device has no render node` failure and loaded `pvr_dri.so`.
This validates the proposed KWin render-node selection fix.

Hardware XWayland acceleration is not complete. The next failure is inside the
PowerVR EGL/glamor path:

- desktop OpenGL context setup reports `Failed to make EGL context current with GL`;
- glamor reports an unexpected depth-30 read format/type and falls back;
- DRI3 cannot create the PVR screen;
- `glxinfo -B` reports Mesa llvmpipe with `Accelerated: no`.

The remaining work is therefore XWayland compatibility with the proprietary
PowerVR EGL/DRI stack, not linux-dmabuf device discovery. The temporary
override, autologin, and swap were removed after the test; the system wrapper
was restored to the `gpu6` configuration and Debian XWayland `22.1.9`.
